> **Имя пакета (Namespace):** @alexbocharov/youtrack-scrum-automation\
> **Целевая платформа:** YouTrack 2021.4+ (Legacy JS API / Совместимость с Рефлексией)\
> **Релиз:** Май 2026

## 🌌 Введение и ключевая философия
В современных инженерных экосистемах ручное управление задачами является узким местом. Высокоскоростная разработка требует, чтобы платформы выступали в роли автономных координаторов, а не пассивных хранилищ данных. Данный пакет воркфлоу превращает YouTrack 2021.4+ в активного оркестратора вашего конвейера поставки.
Связывая текстовые метаданные задач с фазами выполнения стейт-машины (state-machine), этот воркфлоу мгновенно внедряет визуальные индикаторы ([CR], [READY QA], [QA]) прямо в заголовки Agile-карточек. Он динамически распределяет задачи между сотрудниками, выстраивает шлюзы безопасности (гейты) для транзакций и обеспечивает строгое соблюдение этапов передачи — полностью освобождая команды продукта и разработки от административной рутины по маршрутизации задач.

## 🛠️ Системные требования и конфигурация схемы
Перед подключением данного пакета к целевому проекту убедитесь, что схема данных проекта подготовлена и содержит следующие кастомные поля (названия чувствительны к регистру):
## 1. Главный статус жизненного цикла (state)

* Тип поля: EnumField.fieldType (Состояние, одиночное значение)
* Целевые колонки доски: Открыта ──► В работе ──► Тестирование ──► Готово
* Примечание: Начальные фазы («Черновик», «Проектирование») должны оставаться скрытыми из активных представлений спринта на Agile-доске.

## 2. Внутренний этап проверки (ReviewStage)

* Тип поля: EnumField.fieldType (Перечисление, одиночное значение, Обязательно должно быть разрешено пустое значение / Can be empty)
* Варианты значений справочника (Точное совпадение):
* Code Review — Включает маршрутизацию на Техлида и добавляет флаг [CR].
   * Ready QA — Снимает текущего исполнителя, фиксирует в заголовке флаг [READY QA].
   * In Testing — Сигнализирует об активном тестировании флагом [QA].

## 3. Узел распределения ответственности (TechLeadTeam)

* Тип поля: User.fieldType (Пользователь, одиночное значение)
* Область видимости: Кастомная настройка на уровне Проекта. Значение задается в свойствах проекта для определения инженера, выполняющего автоматическое код-ревью. Если поле не заполнено, конвейер автоматически использует стандартный параметр Руководитель проекта (Project Leader).

## Фаза 1: Автоматическая маршрутизация на ревью

* Триггер: Разработчик перетаскивает задачу (drag-and-drop) из колонки В работе в колонку Тестирование.
* Выполнение: Скрипт перехватывает транзакцию, принудительно устанавливает ReviewStage в положение Code Review и добавляет префикс [CR] в начало заголовка задачи.
* Маршрутизация: Карточка автоматически изымается у разработчика и назначается на пользователя, указанного в проектном поле TechLeadTeam.

## Фаза 2: Шлюз контроля (Hard-Gate) Код-Ревью

* Триггер: Техлид проверяет кодовую базу. В случае одобрения он переключает ReviewStage с Code Review на Ready QA.
* Выполнение: Тег в названии задачи обновляется на [READY QA].
* Маршрутизация: Поле Исполнитель автоматически сбрасывается в null (Исполнитель не назначен). Это сигнализирует открытому пулу QA-инженеров о том, что карточка готова к функциональному тестированию.
* Исключение для Багов (Bypass): Если тип карточки — Ошибка / Bug, скрипт обходит публичную очередь тестирования, автоматически возвращает её Автору (тестировщику, который её обнаружил) и переключает этап в положение In Testing ([QA]).

## Фаза 3: Взятие задачи в QA

* Триггер: QA-инженер забирает неназначенную задачу с тегом [READY QA], ставит себя Исполнителем и переключает этап проверки в положение In Testing.
* Выполнение: Тег в заголовке меняется на [QA], блокируя задачу в фазе активной верификации.

## Фаза 4: Завершение и обработка регрессии
В зависимости от результатов верификации запускается один из двух нативных сценариев:

* Сценарий А (Пройдено): Тестировщик перемещает карточку в колонку Готово. Движок полностью очищает текст заголовка (удаляет любые артефакты [QA] или [CR]) и безопасно сбрасывает вспомогательные этапы через операцию delete.
* Сценарий Б (Сбой/Регрессия): Тестировщик возвращает карточку обратно в колонку В работе. Движок запрашивает метаданные истории транзакции (issue.updatedBy), автоматически переназначает карточку на исходного разработчика, очищает ReviewStage и стирает все префиксы заголовка. Карточка появляется в очереди разработчика в своем первоначальном, чистом формате.

## 🔒 Шлюзы безопасности и валидация блокировок
Для предотвращения случайной передачи задач и нарушения порядка выполнения, воркфлоу выступает в роли жесткого системного валидатора:

* Блокировка этапа ревью: Пока задача находится в колонке Тестирование со статусом этапа Code Review, ручные изменения поля Исполнитель запрещены. Если пользователь попытается назначить исполнителем тестировщика или другого разработчика, транзакция будет отклонена. Воркфлоу откатит изменения исполнителя назад на назначенного Техлида и выведет на экран пользователя системное окно с предупреждением об ошибке.

```
+-----------------------------------------------------------------------------------+


| ❌ ТРАНЗАКЦИЯ ОТКЛОНЕНА                                                            |
+-----------------------------------------------------------------------------------+


| Ошибка: Ручное переназначение исполнителя запрещено во время активного Код-Ревью. |
| Текущая фаза: [CR] Code Review                                                    |
| Обязательная роль: Назначенный Техлид проекта                                     |
| Действие: Откат параметров исполнителя для сохранения целостности процесса.       |
+-----------------------------------------------------------------------------------+
```

## 📊 Телеметрические и инженерные метрики
Благодаря реализации четких, ограниченных этапов вместо разрозненных досок и десятков колонок, аналитики данных и руководители ИТ-департаментов могут извлекать прозрачные операционные метрики напрямую из дашбордов YouTrack:

* Обнаружение узких мест (Bottlenecks): Подсчет карточек с префиксом [CR] мгновенно показывает текущую нагрузку на очередь код-ревью без необходимости проведения синхронизационных совещаний.
* Скорость QA-тестирования (Velocity): Измерение дельты времени между выпуском тега [READY QA] и закрытием задачи дает продуктовым менеджерам высокоточную информацию о готовности инкремента к релизу.
* Коэффициент регрессии (Bug Rate): Метрика того, как часто карточки падают обратно в «В работе» с помощью автоматического триггера возврата, наглядно демонстрирует проблемы с качеством кода на ранних этапах разработки.
